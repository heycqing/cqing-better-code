# Module B — 样板优先批量执行协议

> DocDrive 三模块之一。解决：**多分支/多目标批量迁移的遗漏与冗余**。
> 设计原则：先做一个被人工验证的样板，再从样板机械生成每个目标的操作集合。

---

## 为什么每目标独立提示词会失败

**失败 1：遗漏概率随目标数线性增长**

要求作者从记忆或非正式参考中枚举每个目标的特定差异。随目标数增加，遗漏某条差异的概率随之增加。

**失败 2：浪费了目标间的结构相似性**

各目标（分支/实例）共享功能核心，差异只在有限的配置维度上。独立提示词要求对每个目标重新推导"共同部分"，既冗余又容易引入不一致。

---

## 两阶段协议

### 阶段 1：样板执行与人工验证

**输入：** Module A 三份文档（CBDA + SEL + BER）

**执行：**
1. 选择一个目标作为样板（选最接近基线配置、特殊差异最少的那个）
2. agent 按 BER 清单在样板上完整执行改造
3. 产出 **Artifact 1：样板 commit**
4. 人工 review 样板 commit（核验：无禁止文件被修改、BER 全部完成）
5. **阶段 2 在样板 commit 获批前不得开始**

**产出：**

| Artifact | 内容 |
|---------|------|
| 样板 commit | 仅含授权 BER 操作的干净提交 |
| 已验证变更清单 | 每个文件修改的前值/后值记录（diff 样板 commit 得到） |

---

### 阶段 2：目标分支预执行确认清单

**输入：** 已验证变更清单 + CBDA 差异表

**执行：**

1. 对每个目标，从已验证变更清单 + CBDA 差异表机械生成**目标操作集合**
   - 与样板相同的操作 → 标记 `COPY`
   - 目标有特定值的操作 → 标记 `ADAPT`（附目标值）

2. 把操作集合呈现为**预执行确认清单**，提交人工 `[Y/N]`：

```
Pre-Execution Checklist — [目标名称]
──────────────────────────────────────────────────────────────
[ ] [文件路径]    [属性名]     [样板值] → [目标值]    ADAPT
[ ] [文件路径]    [属性名]     [值]   → [值]          COPY
[ ] [文件路径]    [属性名]     [值]   → [值]          COPY
    ... (N entries total)
──────────────────────────────────────────────────────────────
Confirm to proceed? [Y/N]
```

3. 人工确认 `Y` 后，agent 才开始写目标文件
4. 如果人工发现 ADAPT 条目有误 → 修正 CBDA 差异表，不修正 agent 提示词

---

## 协议的关键性质

**差异处理可见性**：每个 ADAPT 操作在执行前对人类可见，不是执行后才发现遗漏。

**冗余消除**：共同部分用 COPY 标记，只需确认目标特有的 ADAPT 条目，无需重新指定全部操作。

**自动化程度**：样板 commit 后，目标操作集合可以机械生成（确定性函数：已验证变更清单 + 差异表）；人类只做两件事：① review 样板、② 确认每目标清单。

---

## 应用范围

Module B 适用于所有"多目标同构改造"场景，不限于多分支：

| 场景 | 样板 | 目标 | ADAPT 来源 |
|------|------|------|-----------|
| 多分支 UI 改造 | 一个券商分支 | 其余分支 | CBDA 差异表 |
| 多页面组件迁移 | 一个页面 | 其余页面 | 页面差异配置 |
| 多租户功能对齐 | 一个租户实现 | 其余租户 | 租户配置表 |

---

## 在 CLAUDE.md 中如何引用 Module B

```markdown
**批量改造使用 DocDrive Module B**：
1. 先在一个目标（样板）上完整实现
2. 人工 review 样板后才开始其他目标
3. 每个其他目标：先生成预执行确认清单 → 等人工确认 → 再执行
4. 不使用"直接对所有目标用同一个提示词"的方式
```
