# DocDrive — 方法论定义

> DocDrive 是 cqing-better-code 的核心方法论。
> 原始论文见 `../ref/docdrive-origin-paper.md`（来自某真实生产项目，已脱敏）。

---

## 是什么

DocDrive（Document-Driven AI Execution）是一套面向**人类主调度 + AI 执行**场景的结构化工作流。

它解决的不是"AI 能不能写代码"，而是"AI 在真实工程任务中为什么不可靠"——具体是三个结构性短板：

| 短板 | 现象 | 代价 |
|------|------|------|
| 范围边界弱 | AI 遇到阻塞时自行扩张解法范围 | 污染共享代码、需要 revert |
| 批量迁移无序 | 多分支独立提示词，遗漏分支差异 | 分支配置错误、返工 |
| 视觉验收缺失 | 结构正确 ≠ 视觉正确 | 上线后 24h 才发现视觉回归 |

## 三个模块

### Module A — 约束文档系统

**解决：** 范围边界弱（agent 自行扩张）

三份文档 + 一个协议，把范围边界从"提示词建议"变成"枚举式契约"：

| 文档 | 作用 |
|------|------|
| **CBDA**（跨分支差异分析文档） | 声明禁止修改区 + 记录跨分支配置差异 |
| **SEL**（范围排除清单） | 声明哪些"关注点"被排除，及其含义 |
| **BER**（授权操作清单） | 列出当前功能的完整授权操作，BER 之外一律不做 |
| **阻塞报告协议** | 遇到无法在范围内完成的操作 → 停止 + 输出结构化报告，不自行绕过 |

操作细节见 `../patterns/module-a-constraint.md`

### Module B — 样板优先批量执行协议

**解决：** 批量迁移无序

不对每个分支重复提示词，而是：

1. 单分支样板先做 → 人工 review 样板 commit
2. 从样板提炼已验证变更清单
3. 用差异表（CBDA）生成每分支预执行确认清单
4. 人工确认清单后才开始执行目标分支

操作细节见 `../patterns/module-b-batch-migration.md`

### Module C — 结构化视觉验证清单

**解决：** 视觉验收缺失

把设计稿转化为可执行的 VVC（Visual Verification Checklist），每条属性必须有代码证据（`file:line + 实测值 + PASS/FAIL`），作为**提交前强制门控**。

操作细节见 `../patterns/module-c-visual-verify.md`

## 后续项目对 DocDrive 的扩展

某权限管理项目（2026-06）在 DocDrive 三模块基础上，增加了两个机制：

### 扩展 1：人工审核门（Human Review Gates）

在 Module A/B/C 的机器执行之上，增加了强制的人工介入点：

1. 文档格式审核（首个规格写完后）
2. 编码总闸（所有规格确认前不写代码）
3. 每功能 BER 执行前确认清单 `[Y/N]`
4. 每功能 VVC 提交前人工复核
5. 推全其他目标（分支/租户）前人工批准

这些不是可选的——未经人工审核不得进入下一步。

### 扩展 2：P0/P1/P2 规则治理

把规则按违反后果分层，防止规则堆积后优先级混乱：

- **P0**：违反即回滚（隔离保护、授权门控等）
- **P1**：违反须修正（变更原子性、文档同步等）
- **P2**：偏好级（表格样式、图标规约等）

详见 `../governance/rule-governance.md`

## DocDrive 与 Harness Engineering 的关系

```
Harness Engineering（理论）
│
│  提供：5子系统框架（指令/状态/验证/范围/生命周期）
│
└─→ DocDrive（实践）
    │
    │  实现：针对人类主调度+AI执行场景的三个具体短板
    │  增加：禁止修改区、阻塞协议、批量迁移协议、视觉验证门控
    │
    └─→ 具体项目（项目级定制）
        │
        │  扩展：人工审核门、P0/P1/P2 规则治理
        │  定制：多租户隔离策略、backend-api.md 单一真相来源
        │
        └─→ 其他项目（继承适配）
```

从理论到实践，每一层都保留上层的核心约束，同时增加该层场景下需要的具体机制。
