## 核心主张

**模型决定写什么代码。Harness 管控何时、何地以及如何写。**

同一个模型（Anthropic Opus），同一个提示词（"构建一个 2D 复古游戏编辑器"）：
- 没有 Harness：20 分钟，$9，产出一个不能用的东西
- 有完整 Harness：6 小时，$200，构建了一个真实可玩的游戏

模型没有变。变的是 Harness。

## 五个子系统

```
┌─────────────────────────────────────────────────────────────────┐
│                        HARNESS 系统                              │
│                                                                 │
│   ┌──────────────┐  ┌──────────────┐  ┌──────────────────────┐ │
│   │     指令      │  │     状态     │  │       验证           │ │
│   │              │  │              │  │                      │ │
│   │ AGENTS.md    │  │ progress.md  │  │ 测试 + lint          │ │
│   │ CLAUDE.md    │  │ feature_list │  │ 类型检查             │ │
│   │ feature_list │  │ git log      │  │ 冒烟测试             │ │
│   │ docs/        │  │ 会话交接     │  │ 端到端流水线         │ │
│   └──────────────┘  └──────────────┘  └──────────────────────┘ │
│                                                                 │
│   ┌──────────────┐  ┌──────────────────────────────────────┐   │
│   │     范围      │  │         会话生命周期                  │   │
│   │              │  │                                      │   │
│   │ 一次一个功能  │  │ 开始时运行 init.sh                   │   │
│   │ 明确完成定义  │  │ 结束时执行清理检查清单               │   │
│   │              │  │ 为下一次会话留下交接说明             │   │
│   └──────────────┘  └──────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘
```

| 子系统 | 职责 | 关键文件 |
|--------|------|----------|
| **指令** | 告诉 agent 做什么、按什么顺序、先读什么 | CLAUDE.md / AGENTS.md |
| **状态** | 追踪已完成/进行中/下一步，持久化到磁盘 | claude-progress.md / feature_list.json |
| **验证** | 只有通过的测试套件才算数；不能只靠"看起来对" | build / lint / 类型检查 |
| **范围** | 一次只做一个功能；不越界；不同时做三个半成品 | feature_list.json / BER |
| **会话生命周期** | 开始时初始化；结束时清理；为下一次留干净重启路径 | init.sh / session-handoff.md |

## 有 Harness vs 没有 Harness

```
没有 HARNESS                           有 HARNESS
==============                         ===========

会话 1：agent 写代码                   会话 1：agent 读取指令
         agent 破坏了测试                        agent 运行 init.sh
         agent 说"完成了"                         agent 只做一个功能
         你手动修复                              agent 在声明完成前先验证
                                                 agent 更新进度日志
会话 2：agent 从零开始                           agent 提交干净状态
         不记得之前发生了什么
         重做已完成的工作              会话 2：agent 读取进度日志
         你再次修复                              准确从上次离开处继续
                                                 你是审查者，不是救援者

结果：你花的时间比                     结果：agent 完成工作，
      自己做还多                              你验证结果
```

## 会话生命周期（标准流程）

```
启动
  1. 读 AGENTS.md / CLAUDE.md
  2. 运行 init.sh（安装、验证、健康检查）
  3. 读 claude-progress.md（上次发生了什么）
  4. 读 feature_list.json（哪些完成，哪些待做）
  5. 检查 git log

选择
  6. 精确选择一个未完成的功能
  7. 只做那个功能

执行
  8. 实现功能
  9. 运行验证（测试、lint、类型检查）
  10. 如果验证失败：修复并重新运行
  11. 如果验证通过：记录证据

收尾
  12. 更新 claude-progress.md
  13. 更新 feature_list.json
  14. 记录仍然有问题或未验证的内容
  15. 为下一次会话留下干净的重启路径
```

## 12 讲核心观点速查

| 讲次 | 问题 | 核心观点 |
|------|------|----------|
| L01 | 为什么强模型在真实任务上仍然失败？ | 基准测试能力 ≠ 工程可靠性 |
| L02 | Harness 到底是什么意思？ | 5个子系统；模型决定写什么，Harness 管控如何写 |
| L03 | 为什么仓库必须是唯一真实来源？ | 如果 agent 看不到，它就不存在 |
| L04 | 为什么一个巨大指令文件会失败？ | 渐进式披露：给地图，不给百科全书 |
| L05 | 为什么长时间任务会失去连续性？ | 将进度持久化到磁盘；从上次离开处继续 |
| L06 | 为什么初始化需要单独的阶段？ | 代理开始工作前，验证环境是否健康 |
| L07 | 为什么 agent 会越界和欠完成？ | 一次一个功能；明确的完成定义 |
| L08 | 为什么功能列表是 Harness 基本单元？ | agent 无法忽视的机器可读范围边界 |
| L09 | 为什么 agent 过早宣布完成？ | 验证缺口：信心 ≠ 正确 |
| L10 | 为什么端到端测试能改变结果？ | 只有完整流水线运行才算真正的验证 |
| L11 | 为什么可观测性应该属于 Harness？ | 看不到 agent 做了什么，就无法修复它破坏的东西 |
| L12 | 为什么每次会话都必须留下干净状态？ | 下一次会话的成功取决于这一次会话的清理 |
