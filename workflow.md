# AI 协作工作流与会话协议（跨项目通用版）

> 版本：v2（2026-06-25）
> 基于 DocDrive 方法论（见 `foundations/docdrive.md`）+ 实际项目实践提炼。

---

## 一、AI 协作分工模型

### 标准三角色

| 角色 | 职责 |
|------|------|
| **Claude** | 审查代码、确认修改点、产出详细编码描述文件、必要时直接改代码 |
| **Codex / 其他 AI** | 按描述文件执行编码 |
| **用户** | 最终验收、拍板决策、给出确认 |

### 关键规则

- Claude 产出的描述文件必须**详细、完整、含当前进度、含模块完整功能列表**，可被其他 AI 直接消费执行，不需要再问问题
- 讨论阶段：列出修改/更正/增加/删减的要点（简要）→ 等用户确认 → 才执行
- 用户说"稍等"/"等一下" → 停下，不输出
- **头脑风暴优先**：需要作出结构性决策时，先与用户确认方向，不提前动文件

## 二、会话启动协议

每次会话开始，按顺序执行：

```
1. 读项目 CLAUDE.md（自动加载）
2. 读 claude-progress.md 的「当前已验证状态」
3. 读 feature_list.json，确认当前 in_progress 功能和优先级
4. 确认验证环境可用（build/lint 是否正常）
5. 确认无遗留 blocker
6. 才开始当前会话的工作
```

**不满足条件时的处理：**
- feature_list 有多个 in_progress → 先向用户报告，确认哪个继续
- 验证环境不可用 → 先修验证环境，不写业务代码
- 有未记录的半成品 → 先补记录

## 三、功能执行协议（结合 DocDrive Module A）

### 开始一个功能

```
1. 读对应规格文档（含 BER / VVC）
2. 向用户列出 BER 执行前确认清单 [Y/N]（DocDrive Module A + 审核门 3）
3. 等用户回复 Y 后，才开始动文件
4. feature_list.json 将该功能置 in_progress（需用户确认写入内容）
```

### 执行中

```
- 只做 BER 清单内的文件和操作（DocDrive Module A）
- 每次接口变更：同一操作内同步 backend-api.md（见 §四 变更原子性）
- 遇到禁止修改区：立即停，输出阻塞报告（DocDrive Module A 阻塞协议）
- 遇到不确定：停下问，不猜测执行
```

### 完成一个功能

```
1. 跑 build 验证（必须通过）
2. 跑 grep createApi URL 核验（必须全部在 backend-api.md 有记录）
3. 向用户展示 VVC 证据（file:line + 实测值，DocDrive Module C）
4. 等人工审核门通过 ← 在此停止输出，不自行继续（改进 #2）
5. 向用户展示拟写入 feature_list.json 的内容 → 等确认 → 写入
   - 有后端 TODO 未清理 → 状态填 pending-backend，不填 passing（改进 #4）
6. 立即更新 claude-progress.md（状态变化时即刻，不等到会话结束）（改进 #4）
```

## 四、变更原子性协议

> 详见 `improvements/doc-sync-v1.md`

**核心原则：** 变更 = 代码改动 + spec 同步。两者缺一，变更未发生。

**文档先于代码**（针对三种变更来源）：

| 变更来源 | 操作顺序 |
|----------|----------|
| 后端通知接口变化 | 先改 `backend-api.md` → 再改 service 代码 |
| 会话中决定换方案 | 先改规格文档 → 再写代码 |
| 直接写代码接入新接口 | 先在 `backend-api.md` 登记 → 再写 `createApi` 调用 |

**机械核验（会话收尾必做）：**
```
grep 项目内所有 createApi/fetch 调用的 URL
→ 逐一在 backend-api.md 查有无对应条目
→ 找不到 = 变更未完成 = 补文档后才能关会话
```

## 五、多目标批量执行协议（DocDrive Module B）

当任务涉及多个同构目标（多分支/多页面/多租户）时：

```
1. 先做单一样板目标（完整 + BER 清单 + VVC）
2. 等人工 review 样板后，才开始其他目标
3. 每个其他目标：先生成预执行确认清单 → 等人工确认 → 再执行
```

操作细节见 `patterns/module-b-batch-migration.md`。

## 六、需求变更协议

当用户提出需求变更（改方案、加功能、改功能、删功能）时：

```
1. 输出影响评估（3 项）：
   - 影响的文件清单
   - 是否需要回滚已完成工作（及工时估算）
   - 新方案与旧方案的关键差异

2. 等用户确认后再执行

3. 执行时：先改规格文档，再改代码
```

不执行"先动代码看看效果再说"的模式。

## 七、会话收尾协议

每次会话结束前，必须完成以下 checklist。未完成项 = 会话未结束：

- [ ] build/lint 通过（视项目而定）
- [ ] grep createApi URL → backend-api.md 全部有对应条目
- [ ] claude-progress.md 已更新（核验准确性，不是首次填写）
- [ ] feature_list.json 真实反映状态：passing = 无残留 TODO；有后端 TODO = pending-backend
- [ ] 无半成品处于未记录状态
- [ ] 会话较长时已补 session-handoff.md

## 八、验证环境降级协议

当理想验证环境不可用时（dev server 无法访问、UI 渲染异常、需要特定账号/数据），不得自行绕过或新增基础设施，应执行以下步骤：

```
1. 停止：不继续写依赖此环境的业务代码
2. 声明：在 claude-progress.md 记录"当前验证环境状态 + 不可用原因"
3. 协商降级方案：
   - 方案 A：用 build 成功 + 代码逻辑核验 作为降级门控
   - 方案 B：推迟此功能，先做不依赖此环境的功能
   - 方案 C：用户提供可用的测试账号/环境后继续
4. 等用户确认选哪个方案
5. 在 feature_list.json 的该功能注记"已降级验证：build 通过，未视觉验证"
```

**核心禁止：**
- ❌ 不得新增 mock 登录、绕过认证、修改 service 来"让环境跑起来"
- ❌ 不得在降级验证下把功能标为完整 passing（只能标 pending-backend 或加注记）

## 八、标准 spec 文件图谱

每个项目应有以下固定名称的 spec 文件。文件名固定，内容按项目填写：

| 文件 | 用途 | 触发更新的事件 |
|------|------|----------------|
| `backend-api.md` | 所有已交付接口的契约（单一真相来源） | 后端新增/修改任何接口 |
| `TODO-backend.md` | 后端缺口集中登记 | 缺口新增/关闭 |
| `feature_list.json` | 功能状态机 | 功能状态变化（需用户确认） |
| `claude-progress.md` | 会话记录 + 当前验证状态 | 每次会话结束 |
| `rule-governance.md` | 规则分层（P0/P1/P2） | 规则新增/变更/退役 |
| `session-handoff.md` | 会话交接摘要 | 长会话/多区域并行时 |

**文件名固定的意义：** Claude 的 CLAUDE.md 和 memory 可以引用固定名称，不需要每个项目记不同的文件名。变化的只是内容，不是结构。
