# 规则治理框架（跨项目通用版）

> 从实际项目复盘提炼，去除项目特定内容后的通用版本。
> 版本：v1（2026-06-25）
> 项目特定版本见各项目 `docs/*/rule-governance.md`。

---

## 为什么需要规则分层

AI 辅助开发项目天然会积累大量规则（架构约束、流程约定、UI 风格、安全边界……）。
如果所有规则平铺堆放，两个问题会出现：

1. **规则冲突时无法判断优先级** — 哪条规则赢？
2. **违反的代价不清晰** — 这条规则重不重要？

分层的目的是：让"违反后果"决定层级，而非让"规则内容"决定。

---

## 三层模型

### P0 — 违反即回滚（Safety）

触碰 P0 = 产出不可信，必须立即停止并回滚到上一个干净状态。

**核心特征**：
- 违反后果影响系统完整性或其他用户/系统（不只是当前功能）
- 必须有机械化检测手段（git diff 范围、grep 特定模式）
- **数量硬限 ≤ 5 条**。超过说明边界不清晰，考虑合并或降级

**跨项目通用 P0 规则**：

| 规则 | 判定标准 |
|------|----------|
| **授权门控**：写代码前必须显式求授权并等批准；规划确认 ≠ 写代码授权 | 未经授权的代码变更 → 回滚 |
| **原型参考边界**：原型（设计稿）只参考交互与样式，不照搬其示例数据/业务逻辑 | mock 数据包含原型种子数据 → 回滚 |
| **不冒充真实**：mock/假接口不得冒充真实通过验证；mock 期不算 passing | feature 标 passing 但仍有 mock → 降级 |
| **隔离保护**（多租户/多环境项目）：不改其他租户/环境的代码路径，不改共用 service 签名 | git diff 涉及禁止修改区 → 回滚 |

> 第 4 条（隔离保护）是项目级 P0，仅在有明确隔离边界的项目中生效。单租户项目可不继承。

---

### P1 — 违反须修正（Quality）

触碰 P1 不需要回滚，但必须在当前功能/会话完成前修正。未修正 = 功能/会话未完成。

**核心特征**：
- 违反后果是质量下降、返工或文档失真
- 可以在当前会话内修正
- 数量建议 5–10 条

**跨项目通用 P1 规则**：

| 规则 | 判定标准 |
|------|----------|
| **单功能流水线**：同一时间只能有一个功能处于 in_progress；功能卡住先解决卡点，不跳到下一个 | 出现第二个 in_progress → 先关一个 |
| **验证闭环**：功能必须经过 build + 运行态验证 + VVC + 人工审核才能标 passing | 缺任一环节 → 不得标 passing |
| **文档代码同步（改进 #1）**：任何接口/字段/行为变更，必须在同一操作里同步 spec 文件；代码改了/文档没改 = 变更未完成 | grep createApi URL 在 backend-api.md 找不到对应条目 → 补文档 |
| **确认前不输出**：等用户明确确认（"确认"/"可以"/"输出"）后，才输出规格或执行代码变更 | 未确认就大段输出 → 停下等确认 |
| **功能清单写入确认**：feature_list.json 写入前向用户展示拟记录内容并等确认 | 未确认的写入 → 撤回 |
| **计划落 MD 文件**：实现计划必须输出到项目 docs/ 下的 .md 文件，不能只停留在对话里 | 计划只在对话中 → 写入文件后继续 |
| **假接口纪律**：独立文件 + 双处 TODO + 契约 .md 描述 + 后端就绪后删除 | 缺任一项 → 补齐后才能继续 |
| **变更影响评估**：需求变更前先输出影响评估（影响哪些文件/是否需要回滚/预计工时），确认后再执行 | 未评估直接执行变更 → 补评估 |

---

### P2 — 偏好级（Preference）

P2 是团队/项目的风格约定，违反不阻塞交付，但应在 code review 时统一。

**核心特征**：
- 违反后果是风格不统一，不影响功能正确性
- 新项目可以选择性采纳
- 数量不限，按领域分组管理

**常见 P2 规则类型（按项目自定义）**：
- UI 风格：组件选型、颜色体系、表格样式基准
- 交互约定：筛选项触发时机、操作列样式
- 代码风格：图标规约、多变体配置化 vs if/else
- 命名约定：文件命名、路由命名、接口命名

## 规则生命周期

### 新增
```
触发事件（踩坑 / 用户指令 / 架构决策）
  → 判定层级（P0 / P1 / P2）
  → 写入本文件对应表格
  → 同步写入项目 CLAUDE.md（P0/P1 必须，P2 按需）
  → 同步写入 memory（feedback 类型）
  → 记录来源（来自哪个项目/事件/日期）
```

### 升降级
- **升级**（P2→P1 或 P1→P0）：违反造成的后果比预期严重 → 升级，记录事件
- **降级**（P0→P1 或 P1→P2）：历史上从未真正触发过问题，约束过严阻碍开发 → 人工批准后降级

### 退役
规则的前提不再成立时退役（项目结束、技术栈变更、业务模式变化）。
不删除，标注 `[RETIRED: 原因, 日期]` 保留历史。

## 规则冲突处理

1. **不同层级冲突**：高层级优先。P0 > P1 > P2
2. **同层级冲突**：走阻塞协议，输出冲突描述 + 候选方案 A/B/C，等人裁决
3. **规则与用户当前指令冲突**：
   - P0：提醒风险并等明确确认，不自动执行
   - P1/P2：用户指令优先，但记录偏差原因

## 跨项目继承指南（新项目启动 checklist）

### 必须继承（团队级，与项目无关）
- P0：授权门控、原型参考边界、不冒充真实
- P1：验证闭环、文档代码同步、确认前不输出、计划落文件、变更影响评估、功能清单写入确认

### 按需继承（项目级）
- P0：隔离保护（多租户/多环境项目才需要）
- P1：单功能流水线（小型项目可放宽）、假接口纪律（有后端依赖时才需要）
- P2：全部清空，按新项目规范重新填充

### 启动步骤
1. 从 `templates/CLAUDE-template.md` 复制到项目根目录
2. 填写「当前活跃任务」和「项目特定规则」区域
3. 从本文件「必须继承」列表复制 P0/P1 规则到项目 CLAUDE.md
4. 确认验证环境可用（**Day 1 优先级，环境不通不写业务代码**）

## 规则审计（每 5 个会话或每个功能完成后）

- [ ] P0 规则数量 ≤ 5？多了考虑合并或降级
- [ ] 是否有从未触发过的 P1 规则？考虑降级
- [ ] 是否有重复/矛盾的规则？合并
- [ ] 是否有新踩坑还没有对应规则？补
- [ ] 所有规则是否都有「来源」记录？补
